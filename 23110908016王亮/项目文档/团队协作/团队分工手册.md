# 🗺️ 山东非遗可视化演示系统：5 人团队分工手册 (V3.0 - 模块化集成版)

> **核心原则**：高度模块化 (High Modularity)。每位成员负责一个独立的逻辑闭环，通过标准接口与主程序集成。拒绝零散的代码修改。

## 🔌 集成与提交规范 (Critical)

为防止“东一块西一块”导致代码无法合并，所有成员必须遵守以下开发守则：

### 1. 代码隔离原则
*   **禁止**直接在 `Form1.cs` 的11111构造函数中写自己的初始化代码。
*   **必须**在自己的 `Form1.XXX.cs` 中创建一个公开的初始化方法，例如：
    *   组员 B -> `public void InitDashboardModule() { ... }`
    *   组员 C -> `public void InitToolsModule() { ... }`
*   **组长**只负责在 `Form1_Load` 中调用这些 `Init` 方法。

### 2. UI 控件命名规范
*   所有控件必须加前缀，防止冲突：
    *   组员 B 的控件：`pnl_Dashboard_...`, `btn_Dashboard_...`
    *   组员 C 的控件：`btn_Tool_...`

### 3. 提交物要求
每次完成一个功能模块，**不只是提交代码**，必须包含可以独立运行的集成说明。

## 📅 开发路线图

1.  **第一阶段 (独立开发)**：
    *   各组员在各自的 `Form1.XXX.cs` 中实现业务逻辑，暂时使用临时的按钮触发测试。
2.  **第二阶段 (接口对接)**：
    *   组员 B 请求 组员 D 提供 `GetAllHeritageFeatures()` 接口。
    *   组员 E 请求 组员 C 提供 `GetNearbyFeatures()` 接口。
3.  **第三阶段 (总装集成)**：
    *   组员 A 回收所有代码，统一在主界面布局，挂载事件。
